# Fraud Risk Intelligence: Anomaly Detection + GLiNER Entity Extraction

**A financial risk analytics prototype combining unsupervised anomaly detection with NLP-based entity extraction to automate SAR pre-population and risk register generation.**

---

## The Business Problem

Financial institutions file thousands of Suspicious Activity Reports (SARs) annually with FinCEN. Each SAR requires:
- Identification of the suspicious transaction(s)
- Structured documentation of key entities (accounts, amounts, geographies, risk factors)
- A narrative description of the suspicious behavior

Today, this process is largely manual. Analysts review flagged transactions, write narratives, and populate structured fields by hand — a slow, expensive, and error-prone process.

This prototype demonstrates a two-stage automated pipeline:
1. **Detect** anomalous transactions using Isolation Forest (unsupervised ML)
2. **Extract** structured risk entities from narrative descriptions using GLiNER (NLP)

The output is a structured risk register ready for analyst review — not a replacement for human judgment, but a force multiplier for it.

```
         ▼

└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│  Risk Register CSV  │  Structured output ready for analyst review
│  Streamlit Dashboard│  Interactive visualization for risk leadership
└─────────────────────┘
```

## Why GLiNER?

[GLiNER](https://github.com/urchade/GLiNER) (Generalist and Lightweight Named Entity Recognition) offers several advantages over traditional NLP approaches for financial risk applications:

| Approach | Limitation | GLiNER Advantage |
|---|---|---|
| Rule-based regex | Brittle, high maintenance | Handles natural language variation |
| Fine-tuned BERT | Requires labeled training data | Zero-shot entity extraction |
| GPT-4 API | Cost, latency, data privacy concerns | **Runs 100% locally on CPU** |
| Traditional NER | Fixed entity types only | **Any entity type via natural language labels** |

For a company managing sensitive financial institution data, local CPU inference with no external API dependencies is a significant operational and compliance advantage.

## Data

This repo uses **synthetic PaySim-style data** generated by `generate_data.py`.

To use the real PaySim dataset:
1. Download from [Kaggle](https://www.kaggle.com/datasets/ealaxi/paysim1) (free account required)
2. Place `PS_20174392719_1491204439457_log.csv` in `data/`
3. Update the path in `run_pipeline.py`

The real dataset contains 6.3M transactions — scale `sample_size` in `gliner_extraction.py` accordingly.

## Usage

### Run the full pipeline:
```bash
python run_pipeline.py
```

### Run individual steps:
```bash
python generate_data.py           # Step 1: Generate synthetic data
python anomaly_detection.py       # Step 2: Score transactions
python eda.py                     # Step 3: Exploratory analysis
python sar_narrative_generator.py # Step 4: Generate SAR narratives
python gliner_extraction.py       # Step 5: GLiNER entity extraction
```

Configuration (paths, thresholds, sample sizes) lives in `config.py`.

### Launch dashboard:
```bash
streamlit run dashboard.py
```

## Risk Register Schema

| Field | Source | Description |
|---|---|---|
| `sar_id` | Generated | Unique SAR identifier |
| `transaction_id` | PaySim | Original transaction reference |
| `risk_score` | Isolation Forest | 0-100 normalized anomaly score |
| `risk_tier` | Derived | HIGH / MEDIUM / LOW |
| `extracted_accounts` | GLiNER | Account IDs from narrative |
| `extracted_amounts` | GLiNER | Dollar amounts from narrative |
| `extracted_countries` | GLiNER | Jurisdictions from narrative |
| `extracted_risk_factors` | GLiNER | Named risk indicators from narrative |
| `extracted_tx_types` | GLiNER | Transaction types from narrative |
| `amount_reconciled` | Validation | Structured amount found in narrative/extraction |
| `country_reconciled` | Validation | Structured country found in narrative/extraction |

## Enterprise Risk Use Cases

Beyond fraud detection, this pipeline architecture applies directly to:

- **Third-party risk management**: Extract entities from vendor contracts and due diligence documents
- **Regulatory document analysis**: Parse examination reports, enforcement actions, regulatory guidance
- **Counterparty risk**: Identify and classify entities across financial institution agreements
- **Operational risk incidents**: Extract structured data from incident narrative descriptions
- **KRI automation**: Auto-populate risk register fields from unstructured operational reports
